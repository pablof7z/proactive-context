# Wiki Architecture

## Spec Authoring

The wiki is designed as a convergent distillate, not a transcript dump, meaning the triage and route stages intentionally shrink volume by dropping redundant sessions and folding claims onto existing statements.

Rejected alternatives should be preserved in the spec with reasoning ('why not'), not just conclusions, to prevent re-litigating settled decisions.

Entity-definition guides (hook-contract, cli-surface, db-schema, config-reference) cannot be mined from conversation and must be hand-authored from the codebase.

The wiki has two structurally different knowledge layers: Layer 1 (decisions, gotchas, corrections — mineable from transcripts because participants state them) and Layer 2 (entity definitions — must be authored directly because participants never state what they both already know).

The project's #1 EXTRACT failure mode is over-splitting claims, which fragments the downstream ROUTE stage (topic→page) — the primary quality bottleneck; widening what counts as spec is desired but pushing finer per-claim granularity is dangerous.

When a session is question-dominated with no spec delta, the probed topic is routed to capture_research/docs/wiki/research/ as a research/topic seed to capture that the user cared about the topic.

Topic routing uses metadata-only topic fields on flat files rather than physical `topics/<topic>/<slug>.md` directories, to avoid touching 10+ call sites and colliding with the podcast `docs/wiki/topics/` directory.

The ROUTE altitude prompt change makes ROUTE assign one coherent topic per guide and drops the bogus guide-count target that was causing slug fragmentation.

The `topic` field is added to `GuideFrontmatter` and `IndexRow`, parsed from frontmatter, and serialized only when non-empty for backward compatibility.

RERANK receives full-catalog context grouped by topic before per-claim candidates, so it can assign claims to existing topic areas instead of minting blind.

RERANK emits a `topic` field per route decision, and existing guides get their topic back-filled on their next update.

The `_index.md` file is rendered as a topic-grouped table with one section per topic and a 7th column for topic, human-readable and machine-parseable.

The `wiki-publish-cleanup` branch (ROUTE altitude fix, wiki tidy, normalize_for_publish, wiki doctor) was merged into master.

Derived wiki cache files (_index.md labeled 'rebuilt after each capture' and _citations.log) are gitignored to avoid churn/conflicts from concurrent agent captures, while the 9 wiki guide .md files are tracked. The `docs/wiki/` directory is untracked/generated state that does not travel with the branch; completed wikis exist only on the originating machine, and concurrent peer sessions can regenerate the live wiki mid-run, confirming the multi-agent shared-state hazard. The Phase 0 baseline doc includes a volatility caveat noting that the generated wiki is untracked and was regenerated by a peer session mid-session, so it must not be relied upon as stable during multi-agent work. .DS_Store files are gitignored tree-wide.

Guides are stored at <wiki>/guides/<slug>.md instead of <wiki>/<slug>.md. (Previously: guides were stored at the wiki root.) Guide path resolution falls back to the legacy root location for not-yet-migrated wikis, eliminating a flag-day migration requirement. An idempotent migration function (`migrate_guides_to_subdir`) moves flat root `*.md` guides (skipping `_index`/`_citations`) into the `guides/` subdirectory, and is run by `rebuild_index` before scanning.

## Retrieval Integrity

Stale or contradictory wiki headings can poison inject retrieval when the heading chunk is surfaced instead of the body, because the heading is treated as authoritative content. For example, the `capture-v04-pipeline-status.md` wiki doc had a stale section heading stating v0.4 was not built while the body said it was live; inject retrieved the heading chunk and produced a wrong answer, so the heading and summary were corrected to reflect that v0.4 is implemented. <!-- [^d00d6-9] -->

## Topic Routing

Topic routing uses metadata-only topic fields on flat files rather than physical `topics/<topic>/<slug>.md` directories, to avoid touching 10+ call sites and colliding with the podcast `docs/wiki/topics/` directory. The wiki deliberately uses a slug-flat storage architecture where a guide's path identity is decoupled from its mutable topic attribute, grouping guides only in the derived `_index.md` view rather than in topic subdirectories. Coupling a mutable attribute (topic) to a file's identity (its path) introduces fragility because topics change, meaning files must move via `fs::rename` on every re-topicing operation rather than simply updating one frontmatter line. Under a topic-folder layout, relative links like See-Also (`[Name](slug.md)`) and `_index.md` table hrefs break on every topic change, requiring all cross-topic inbound links to be rewritten when a file moves to a new directory. Under a topic-folder layout, slug-to-path resolution stops being O(1) and would either require threading the topic through all callers or performing a recursive `**/slug.md` search, since many callers (e.g., `inject.rs:700`, capture loaders) only have a slug. Under a topic-folder layout, the four non-recursive `read_dir` calls in `enforce_bidirectional_links`, `rebuild_index`, `read_index_live`, and the statusline would silently return zero guides unless converted to recursive walks. Under a topic-folder layout, concurrent capture by peers introduces a new race condition class where file renames race with concurrent reads/writes of the same slug, a failure mode the flat layout does not have. Under a topic-folder layout, every re-topicing reshuffles files across folders, producing large rename diffs and more merge conflicts when peers edit the same guide. In the flat design, re-topicing is a one-line metadata edit with zero file moves, zero link rewrites, and zero resolver changes, because `topic` is a pure attribute and `_index.md` is a derived view. <!-- [^d88b0-2] -->

## Wiki Architecture

Project wiki regen order is: hl (canary), then tenex-edge, then podcast-player, then nostr-multi-platform (last, biggest). Wiki regenerations must run one project at a time (not in parallel) to avoid overwhelming the LLM. The wiki regeneration for each project must delete `docs/wiki` and reset capture markers to the beginning via `--reset` before regenerating. The wiki regeneration runner script is at `~/.proactive-context/regen_wikis.sh` with abort-on-failure behavior and `--reset --yes` for scoped runs (no `--project`/`--yes` conflict). Three of four wiki regenerations completed successfully: hl (40 guides, 11 episodes, 6 research), tenex-edge (84 guides, 96 episodes, 2 research), podcast-player (98 guides, 71 episodes, 8 research). The nostr-multi-platform wiki regeneration was intentionally parked (not abandoned) at ~104/789 sessions; it can be resumed from that point rather than restarted. The user explicitly said to skip it on the current machine and instead prepare a handoff for another computer. All NMP Claude transcripts (228 sessions across 6 dirs) and Codex rollouts (538 cwd-matched) have been rsync'd to `pablo@157.180.102.242` so the work can continue there.

Inject has no transcript-triage gate; its fast model is a wiki navigator, not a 'should we even look?' gate. This is a deliberate architectural asymmetry vs. capture.
