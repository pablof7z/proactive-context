# proactive-context deep audit — 2026-07-02 (FINAL)

Multi-agent audit (10 subsystem auditors; every major/critical finding independently verified by 2 adversarial reviewers). Buckets: CONFIRMED = both verifiers reproduced it from source; REFUTED = at least one verifier disproved it; minor/trivial findings were not adversarially verified.


## capture-core

### Big picture

The core capture pipeline (EXTRACT → mechanical authority tagging → ROUTE recall+rerank → per-guide RECONCILE) is conceptually strong: Rust owns line-numbered evidence slicing, authorship attribution, citation minting, and receipt-after-mutation ordering, so the model can never fabricate a citation or an author. The staged design, the tag-don't-drop authority model, and the seeds/episodes/research/nouns partitioning all match the specs closely. The epistemics are the healthiest part of the subsystem.

Reliability engineering has not kept up with the epistemics, and this is where the subsystem is heading wrong. The dominant pattern is "degrade silently, then declare success": every LLM/JSON failure in every stage is coerced to an empty value (`unwrap_or_default`), the run proceeds, and the session is then permanently marked captured regardless of outcome — including timeouts and hard errors. Since delivery-side under-injection is the declared cardinal failure and capture is its upstream, a capture pipeline that silently converts transient API failures into permanent, retry-suppressed data loss violates the project's own north star. Worse, the error paths recreate the exact failure modes the design docs identify as solved: a single unparseable ROUTE response re-creates the over-split ghost-guide catastrophe (one junk guide per claim), and the destructive `remove` op is the only op exempt from evidence verification — inverting keep-everything, the one invariant the specs repeat most.

The concurrency story is half-built. Guide mutations correctly take a project write-lock with re-read-inside-lock, but the lock key is derived from the raw hook cwd rather than the resolved project root, so two sessions in different subdirectories or worktrees of the same repo (the documented multi-agent workflow) take different locks on the same wiki. Structural maintenance then rewrites every guide in the wiki with no lock at all, from a snapshot taken before it started, silently clobbering concurrent RECONCILE writes. All guide writes are non-atomic `fs::write`, and the scheduler unconditionally SIGTERMs a possibly-mid-write predecessor. Finally, flag/variant drift is accumulating: `build_extract_system` documents itself as the single source of the production EXTRACT prompt, but the live path bypasses it, leaving the "on by default" granularity block and the entire PC_EXTRACT_VARIANT/PC_CLAIM_STATUS machinery dead in production while tests assert they are live. The debug tooling and the production pipeline are quietly diverging — the exact failure the shared-triage-function comment says the codebase learned to avoid.

## episode-research-capture

### Big picture

The episode/research capture subsystem is conceptually strong — both stages implement the "recognition + provenance, not synthesis" thesis faithfully, evidence ranges are Rust-verified, immutability is enforced on the live paths, and the supersedes linker plus status patches respect keep-everything (bodies never rewritten, only frontmatter metadata). The spec documents are unusually honest and the code matches them in most places. But the subsystem has grown as a chain of near-identical sibling stages (research → episode → nouns → realness) bolted after the main capture pass, each with its own copy of transcript parsing, its own idempotency story, and its own filename scheme — and the copies have already diverged in ways that matter.

The single biggest incoherence is at the delivery boundary: research capture is ON by default (an LLM call per session, records written to docs/wiki/research/), but inject only surfaces research records behind an env flag (PC_RESEARCH_CATALOG) that defaults OFF and that nothing in production sets. Under the project's own doctrine — delivery is system-driven push, under-injection is the cardinal failure, R4 says inject delivers findings with method attached — the default configuration makes research records write-only memory. Worse, the typed SELECT preamble (default ON) tells the selector about `research:` keys that never exist in the catalog it sees.

The second theme is that the two sibling stages have opposite and both-wrong idempotency semantics against capture's grow-and-recapture model (a session re-captures whenever its exchange count grows). Episodes gate recognition once-ever-per-session, so decisions made in the later half of a resumed session are structurally invisible — permanent under-capture justified by a comment that is factually wrong. Research has no gate at all, so recognition re-runs on every recapture and dedups only by a collision-prone filename (`<date>-<idx>-<slug>`, no session component), which simultaneously produces duplicates (model rephrases the characterization) and silent data loss (two sessions on the same date colliding, second arc dropped forever while being reported as persisted). Research also ignores the archeologist date_override — a bug the episode spec explicitly names ("a latent backfill bug there") — so historical backfill stamps every record with the backfill date, maximizing collisions and breaking the R5 chronology the record chain is supposed to embody.

Where it is heading wrong: each new knowledge type re-implements transcript parsing (transcript.rs, research_capture.rs, episode_capture's build_dialogue, plus a dead merged_recognition prompt copy), so harness-format drift must be fixed N times — the task-notification extractor is already a verbatim duplicate in two files, and the array-form-content blindness is latent in both. Meanwhile the episode card's provenance contract quietly degrades at write time: evidence line numbers index an ephemeral in-memory flattening, the verified slice text is discarded, and the frontmatter points at the raw JSONL path those numbers don't apply to. Consolidating parsing into transcript.rs, adding a session component to filenames, and wiring research delivery (or turning capture off) are the three highest-leverage fixes.

## inject-gate

### Big picture

The inject subsystem is architecturally mature and mostly coheres with the push-at-critical-points doctrine: a UserPromptSubmit hook runs cheap retrieval, a fast SELECT gate over a typed catalog, a strong COMPILE pass with a per-session dedup ledger, and an LLM-free noun resolver that genuinely operates as a peer layer (every terminal arm — trivial prompt, retrieval failure, no API key, short-circuit, compile error, timeout — falls back to `commit_noun_only`, which is exactly the right shape). The degradation ladder is deliberate and legible.

Three structural tensions are pulling it in the wrong direction. First, latency-budget accounting is unowned: the 25s inner timeout only wraps SELECT+COMPILE, while unbounded pre-work (ONNX embedder load, vector search, transcript parse) runs before it, under a 30s hard harness kill — so the worst-case path is killed by the harness with zero output AND zero telemetry, the exact silent under-injection the project calls its cardinal failure. Several other failure modes (config load error, missing DB) also bail before event-context init, so the system cannot see its own under-injection. Second, the taxonomy expansion (research/nouns/claims as catalog rows) was half-shipped: the SELECT preamble that tells the gate it is REQUIRED to pick [research-record]/[claim]/[noun-entry] rows is default-ON, while the flags that put those rows in the catalog are default-OFF, and the COMPILE citation-labeling code was never taught the new key prefixes — so the moment the hybrid actually turns on, every citation from those sources is a dead path. The citation contract in general is prompt-enforced only; nothing ever verifies an emitted (path:line) exists.

Third, the cross-turn dedup ledger models "already injected ⇒ still visible" as a session-lifetime invariant, but Claude Code compaction breaks that premise while keeping the session_id — after compaction the compiler is explicitly instructed to suppress facts the assistant can no longer see. Given that the ledger's stated failure preference is "re-inject rather than suppress", this blind spot fails in the forbidden direction. Smaller frictions compound it: fallback (raw-chunk) injections are never ledgered so degraded turns re-dump identical chunks, and the raw-chunk fallback itself is the rejected context-dump delivery shape kept alive as the primary degradation mode.

Where it is heading wrong: features land as env-flag arms (PC_* flags read process-wide, tested via global env mutation) whose defaults drift out of sync with the prompts that reference them, and every error path swallows into `Ok(())`. The subsystem needs a single budget/deadline object threaded through the hook, a citation-label resolver shared with `read_catalog_content`, and an inject health signal (a `pc doctor`-style check that the last N prompts actually produced inject.done events) — otherwise under-injection regressions will keep being invisible by construction.

## wiki-doctor

### Big picture

The wiki store has a healthy deterministic core (hand-rolled frontmatter round-trip, citation-marker machinery, derived _index.md, dry-run-by-default maintenance tools) and the "embeddings propose, LLM confirms, code guarantees" discipline in doctor.rs is genuinely good engineering. But the subsystem's stated invariants — KEEP-EVERYTHING, mandatory citations, supersession-links-plus-demotion — are enforced unevenly: doctor's merge path deterministically guarantees citation survival while its sibling cross_supersede deletes markers with no guard at all, and the only true "demote-not-delete" implementation is for episode cards (episode_capture's supersedes linker), not guides. For guides, deletion is real: doctor --apply hard-removes absorbed files and its merge prompt instructs the LLM to drop contradicted statements. The demote-not-delete promise from the staleness plan is aspirational in this code.

The deepest structural problem is that supersession has no durable representation in the guide layer. GuideFrontmatter (wiki.rs:17-31) has no supersedes/superseded_by/status field, and the parser silently discards unknown keys, so no other writer can even add one without it being erased on the next save. Cross-guide supersession is therefore recorded as free prose breadcrumbs ("superseded — see slug.") that the rest of the system cannot see: doctor's remap_see_also won't rewrite them after a merge renames the target slug, bidirectional-link enforcement doesn't traverse them, and cross_supersede itself can't tell it already revised a statement, so re-runs nest breadcrumbs and — because stamp_updated bumps the whole guide's date — every sibling statement in a revised guide masquerades as "newer" on the next pass. Two maintenance mechanisms (merge and cross-supersede) actively degrade each other's outputs.

Concurrency is the second systemic weakness. Capture correctly serializes guide mutations behind a project wiki lock with re-read-inside-lock, but then run_structural_maintenance calls enforce_bidirectional_links outside that lock, and enforce_bidirectional_links unconditionally rewrites every guide in the wiki from an in-memory snapshot — a wide lost-update window against any concurrent session's capture. Doctor --apply and cross-supersede APPLY write with no lock at all, and every writer uses plain fs::write (truncate-in-place), so the inject hook — the system's whole reason for existing — can read a torn _index.md and silently under-inject, the cardinal failure.

Where it's heading wrong: maintenance tooling keeps accreting as parallel one-shot passes (doctor merge, retopic, cross-supersede, taxonomy backfill) that each re-implement load/decide/save with slightly different integrity guarantees, instead of converging on one write path with a shared lock, atomic writes, an open frontmatter schema, and first-class supersession/demotion metadata. Until the schema can carry status/supersedes fields and every writer goes through the locked choke point, each new pass adds another way to lose or contradict knowledge.

## nouns-realness

### Big picture

The noun layer is in a healthier architectural state than its history suggests: the 2026-06-25 peer-resolver rework genuinely landed. Production inject (src/inject.rs) calls resolve_noun_primer up front, independently of guide selection, and every terminal arm (trivial prompt, retrieval failure, no API key, empty briefing, short-circuit, compile error, timeout) attempts a noun-only injection — the "peer, not garnish" model is real. The population is the authority store (nouns/*.md, written only by the C1 definitional stage in capture.rs), guide-title sourcing was verifiably eliminated from the population path (C3 derive_registry survives only as definition ENRICHMENT plus dead/eval code), and the empty-realness-registry-is-inert invariant is tested. The confidence-gated surfacing policy (suppressed never; high-confidence needs real|has-def|direct-query; lone token needs real) matches the stated design exactly.

The main conceptual tensions are at the seams the rework did not finish. First, the layer's own goal statement — "'what is X?' must inject the captured definition" — is defeated by its first-mention gate: the recent-turns suppression reuses the full (including lone-token, low-confidence) matcher over text that includes assistant turns, so the exact motivating prompt injects nothing whenever X (or one shared token of X) appeared anywhere in the recent window. That is under-injection, the project's self-declared cardinal failure, coded as a deliberate "precision leaning". Second, the alias story is split-brain: inject-time matching has a compact-key recall layer (purplepag.es == purplepages), but capture-time accumulation keys the ledger by canonical_key only and never runs cluster_nouns — the very fragmentation alias.rs was written to kill still fragments the live ledger for domain/punctuation spellings, keeping real nouns below the +3 promote threshold. Third, the whole pre-rework primer stack (build_inject_primer, detect_first_mentions, realness_gated_registry, noun_referenced_in) is still compiled, documented as "the single entry point inject.rs calls. Always on.", and embodies a different, ungated matcher — two matchers drifting exactly as feared.

Operationally, the realness ledger — the one store whose contents cannot be regenerated cheaply and whose suppression lines are the anti-confabulation guarantee — is the least protected file in the system: read-modify-write with no project lock (capture holds only a per-session lock) and a non-atomic fs::write overwrite, read on the inject hot path. Meanwhile decay/dormancy exists only as eval code; production realness is monotone-forever and realness.jsonl accrues a permanent zero-score line for every neutral-mentioned candidate. The direction to correct: finish the migration (delete the dead primer stack), give the ledger the same wiki-lock + atomic-rename discipline other wiki writes get, unify aliasing between capture and inject, and make the recent-turns dedup confidence-aware so direct entity queries always deliver.

## recall

### Big picture

The recall subsystem is a competently ported prototype (experiments/recall → src/recall) whose core bet — "load EVERYTHING into a real 1M-context model, recall guaranteed by reading everything" (src/recall/mod.rs:1-4, store.rs:1-2) — is already under strain and has no guardrails around its single load-bearing assumption: that the corpus fits the selected model's context. The code hardcodes num_ctx=1,050,000 for every model (ask.rs:97,105), the picker knows each model's real context but the REPL never compares it to the corpus estimate, and the Ollama path will silently truncate rather than error. Given the project's own measurements (glm-5.1:cloud is 203K not 1M; the clean human corpus already measured at ~2.16M tokens vs the 0.74M the module doc still claims), the "reads 100% of your history" promise can silently become "reads an unknown fraction" — the exact under-recall failure the project calls cardinal.

The second structural tension is that several mechanisms quietly fight the subsystem's two stated integrity guarantees. Time/supersession awareness (ask.rs TIME_RULES) is undermined by corpus dedup, which keeps the first occurrence in uuid-sort order and discards later restatements' dates — destroying the recency signal the prompt tells the model to reason over — and mislabels codex's same-session double-logging as "also said in N sessions". Citation integrity (the validated 100%-citations bar) is overstated by a tolerant resolver that counts any citation with a real project+session prefix as valid regardless of line number, and is entirely absent in the new wiki-oracle mode. The gate adds a third: a cheap-LLM DROP/CLEAN verdict is cached forever, keyed only by id with no text-hash or model/prompt versioning and no guard against model chatter, so one bad cheap-model day punches permanent, invisible holes in a corpus that claims completeness.

Prototype-to-production seams are visible and consistent in direction: freshness is manual (`pc recall index` is never auto-run by ask/repl, directly contradicting the "freshness invariant (critical)" doctrine route_recall.rs states for the sibling embedding recall); coverage is claude+codex only while the archeologist already ingests opencode and tenex; the REPL grew a "fast gate" model selector that is pure dead UI; and recall/llm.rs is yet another parallel OpenRouter/Ollama client that bypasses the event-log/sidecar/usage plumbing the rest of the codebase was just unified onto (commit 9ae027f). None of these is fatal alone, but together they mean recall's confident, well-rendered answers rest on a corpus whose completeness, freshness, and citation validity are all softer than the UI claims. The highest-leverage fixes are cheap: a corpus-vs-model-context preflight, dedup that preserves latest-date metadata, strict citation resolution, gate-cache versioning, and auto-incremental indexing at REPL start.

## providers-config

### Big picture

The LLM-provider layer has fragmented into five parallel client implementations that each reinvent transport, retry, timeout, and accounting policy: openrouter.rs chat_once (async, no retry, 120s timeout, events+sidecar), recall/llm.rs (blocking, 6-attempt 429/503 backoff, 900s, no events), capture.rs call_model_blocking (blocking, 3 attempts w/ Ollama-404-evicted retry, no events/usage), realness.rs (blocking, PC_T0_RETRY=5, its own char-count metering atomics), and rig-core for Ollama in inject/capture (usage invisible, wrapped by record_external_turn with zeroed usage). recall/llm.rs::chat is the one true provider-routed abstraction, but inject and capture do not use it — they copy-paste a three-arm provider match at every call site. The result is that the same provider behaves differently per call site, and the recently-landed crate::usage Ledger — whose module doc claims it now "covers all providers... all call paths" — is in fact recorded from exactly one place (recall repl), and even there the gate call's usage is discarded. Usage accounting is aspiration, not architecture.

The most product-threatening incoherence is in the claude-cli hot path versus the 25s inject budget. The warm pool's design decays to cold: pre-warm covers only the capture model, refill happens only after a successful request, and the reaper kills 5-minute-old children without respawning — so in realistic usage (prompts more than 5 minutes apart, or select model ≠ capture model) every inject pays the ~30s node boot the pool exists to amortize, against a hardcoded 25s per-call timeout. Failed attempts spawn-and-kill more cold children instead of building warmth. For the provider that configure ranks first, this structurally produces the cardinal failure the project defines for itself: under-injection via timeout fallback.

Observability — the subsystem's stated reason for existing (openrouter.rs replaced rig "so every request/response can be logged") — is quietly broken in three compounding ways: the known sidecar-overwrite bug is still live (run_stage hardcodes turn=1, so EXTRACT/ROUTE/every-RECONCILE share one filename per capture session); chat_once never sends usage:{include:true} so OpenRouter cost/cached-token data is absent from the hot paths (recall/llm.rs proves the team knows the flag is required); and the just-shipped claude_cli.call tail event logs the full system+user prompts into an event pipeline that silently drops any line over 4095 bytes — so it never lands for real calls, and leaks prompt text into the shared log when it does. Meanwhile the TUI reads payload["cost"] where the event writes "cost_usd". Each layer was built against an imagined version of the adjacent layer.

Where it is heading wrong: every new feature (research capture, episode cards, delta-extract, edge linking, realness) grabs whichever of the five clients is nearest, further entrenching the fragmentation. The fix is boring and known — make recall::llm::chat (returning Reply{content, usage}) the single provider entry point, thread a &mut Ledger/event hook through it, and give the claude sidecar a warmth policy driven by the configured inject roles. Also worth noting: the working tree replaces the just-committed ratatui provider-tabs configure with a simpler inquire rewrite (uncommitted −1016 lines), and config.rs defaults have drifted from both their own doc comments and recall/mod.rs's intended defaults — config-as-written wins and ships a different answer model than the code says it defaults to.

## infra-daemon

### Big picture

The subsystem is a plausible sketch of a local-first indexing pipeline (per-project fork-daemon + sqlite-vec index + JSONL event bus + declarative harness installer), and several hard lessons have clearly been absorbed (content-hash skip, watch-dir exclusions, PIPE_BUF-atomic event appends, idle-exit). But its process-lifecycle and concurrency story is built entirely on check-then-act filesystem idioms — pidfile read-then-rename, `.indexing` lock exists-then-remove-then-rename, signal-based liveness via SIGCONT — none of which are actually atomic or side-effect free. Every multi-process scenario the design itself creates (hooks firing concurrently in two Claude windows, a daemon plus a hook, an old daemon plus an upgraded binary) can produce duplicate daemons, stolen locks, orphaned watchers, or an innocent process getting signaled. The sqlite layer compounds this: there is no busy_timeout anywhere, and `open_db` is a writer even on read paths, so the moment two processes genuinely overlap, the failure lands on the inject hook — which swallows errors and injects nothing. Under-injection, the system's self-declared cardinal failure, is the default failure mode of this subsystem's races.

The deepest conceptual crack is that index freshness has no owner. The daemon idle-exits after 6 hours on the promise that "a new request re-spawns one on demand," but the only respawn trigger is `!db_path.exists()` — which is false forever after the first index. Meanwhile `init_schema` will silently DROP the whole vector table on any dim/schema-version skew at any open (including a hook's), and nothing ever schedules the full re-index that wipe requires. So the two mechanisms fight: one retires the only process that could heal the index, the other quietly destroys the index from processes that cannot heal it. The result is a system that trends toward a stale-or-empty index that still reports success, starving the push-delivery loop the whole product is built around.

There is also an inverted use of the sidecar architecture: the long-lived daemon (which could amortize model load itself) delegates embedding to the shared sidecar, while the latency-critical per-prompt inject hook cold-builds a full in-process ONNX embedder on every UserPromptSubmit — exactly backwards relative to the 25s hook budget. And the observability layer undercuts itself: `log_event` silently drops any event over 4095 bytes while `claude_cli.call` logs full system+user prompts, so the events most worth seeing (and that `pc tail`'s new rendering was just built for) mostly never reach the log. The harness installer is the healthiest part (genuinely declarative, idempotent per-path), but it will replace an unparseable settings.json with `{}` and identifies "our" hooks by the current binary path, so path changes accumulate duplicate hook registrations — duplicating the whole inject pipeline per prompt.

## integrations-archeologist

### Big picture

The archeologist's core architectural decision is right and mostly executed: it does NOT reimplement extraction — every session funnels through run_capture_for_archeologist → run_capture_from_input (src/capture.rs:2926-2944), the exact staged TRIAGE→EXTRACT→RECONCILE pipeline the live hook uses, with today_override providing historical dating and skip_maint/checkpoint batching as the only behavioral deltas. The drift risk the design feared shows up in a narrower, subtler place: the archeologist parses transcripts with parse_transcript_meta and strips isSidechain/isMeta turns by default, while the live hook uses parse_transcript and keeps them — so the same transcript captured live vs. backfilled yields different claims and different explicit/implicit authority tags. The codex/opencode/tenex sources are handled well at the format level (codex rollouts parse natively through the shared decoder; opencode/tenex synthesize flat JSONL the pipeline already understands), but they are second-class citizens operationally: invisible in the default interactive picker, un-resettable per-project, and their per-source ProjectInfo grouping breaks the oldest-first chronology guarantee that RECONCILE's latest-wins semantics depend on.

The subsystem's biggest conceptual weakness is its state model. A single marker file per session (captured-sessions/<id>.json) is asked to mean three different things — "captured successfully", "capture attempted but failed/timed out", and (by omission) "triage said skip" — and it gets all three wrong for a bulk backfill: failed and timed-out sessions are permanently marked captured (silent knowledge loss feeding the cardinal under-injection failure, since the staged pipeline's chat_once has no 429/5xx retry and the archeologist hammers sessions back-to-back), while triage-skipped immutable historical sessions are never marked and re-pay a triage LLM call on every re-run. The interactive picker's routing_cwd merge compounds this: sessions captured into the wrong project's wiki are globally marked captured and can never be captured into their own project without a full reset.

The second systemic weakness is that the run's ground truth is a side-channel: counters, cost, and the feed are reconstructed by tailing events.jsonl and joining on a project key that is computed two different ways (normalize_path(cwd) at event-emission vs normalize_path(resolve_project_root(cwd)) at scan time), so worktree/subdirectory sessions — common in this user's workflow — silently produce a blank dashboard and misreported "too-short" counts. This is the third bespoke tailer in the codebase (tail.rs, tui.rs, archeologist.rs), already drifted (no truncation handling). Finally, there is a ring of dead-but-documented machinery — run_episode_repair_for_session (the documented re-run convergence path, never called), collect_run_counters, run_picker, and a --jobs flag that advertises parallelism but only toggles the TUI off — that makes the subsystem look more finished than it is; the "live run-view TUI was stubbed" note in memory is stale (the TUI is real now), but the resume/idempotency story it implies is not.

## hygiene-coherence

### Big picture

The core loop (capture -> wiki -> inject) is genuinely coherent with the stated principles: push-only delivery via UserPromptSubmit hooks, citation-by-construction (models point at line numbers, Rust slices), authority tags as metadata-only, and the noun/realness peer layer wired into inject exactly as designed. Most of the 8 confirmed stress-test bugs were actually fixed (the fresh-project citation-log ordering, the multibyte truncation panic via `tail_capped`, orphan citations gated on `!result.starts_with("Error:")`). The e2e bash tests even assert the metadata-never-rendered principle directly. This is a system whose principles are mostly enforced in code, not just in docs.

The biggest conceptual crack is at the exact spot the principles say should be safest: the RECONCILE `remove` op physically deletes an entire guide section (`guide.body.replace_range(start..end, "")`) with no supersession breadcrumb, no demotion, no event log entry — and it is the ONLY mutation op for which Rust-verified evidence is explicitly waived (`if op.op != "remove" { continue }`). Under KEEP-EVERYTHING and integrity-by-construction, the one destructive operation is the one operation a hallucinating model can invoke uncited. Adjacent to this, two stress-test bugs that threaten the store survive unfixed: failed/timed-out staged captures are still permanently marked captured (silent knowledge loss on any provider outage), and structural maintenance (which re-serializes every guide) still runs outside the project wiki lock while every guide rewrite silently drops unknown frontmatter keys.

Structurally, the repo is drifting toward two kinds of entropy. First, frozen experiment mass: ~9,550 lines of eval_run7..15/t0/realness/prompt_variant code (~20% of src) is compiled into the shipping `pc` binary, some of it with Pablo's absolute home paths hardcoded — it can never run anywhere else, yet it pins production internals (`run_structural_maintenance_for_eval`, route_recall, realness) and generates most of the 42 compiler warnings. Second, duplication: there are at least 5 independent Claude-JSONL transcript parsers, 4 hand-rolled LLM provider-dispatch HTTP stacks (each with its own retry/think/timeout semantics), 4 frontmatter parsers, and 4 extract-JSON-from-LLM-output helpers. This duplication has already produced principle-relevant divergence: the subagent-task-result blindness fix (PC_INCLUDE_TASK_RESULTS) lives only in transcript.rs's canonical parser; promptSource-based human-utterance detection lives only in episode_capture; each transport re-solved retries separately. Every future capture-pipeline fix must now be applied N times or it silently doesn't apply to some artifact type — a direct threat to "routing is the capture bottleneck" work.

Docs and repo hygiene lag the code by weeks: README advertises a removed feature (`pc agents`) and four renamed commands; `pc --help`'s about-string still describes the pre-pivot "RAG over markdown" product; CLI help asserts flags default OFF that default ON. A 1,646-line uncommitted diff (full configure.rs rewrite) sits on master — dangerous given the documented multi-agent branch-GC hazard — next to an untracked 2.1MB docs.zip with __MACOSX cruft, an untracked-but-not-ignored docs/wiki/, and no CI at all.
